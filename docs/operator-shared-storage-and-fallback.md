# OperatorApp Shared Storage and Fallback Notes

## Overview

This document summarizes the shared storage surfaces that coordinate the operator tool with other parts of the system and records how the application falls back to the legacy processing path when newer signals (such as a completion flag) are absent. Use it as a reference when extending the embed handshake or adding new shared fields.

## 2. Fallback when the completion flag or preflight context is absent

When the embed host does not expose a completion flag (or the display session omits the new field), or when no cached preflight context is available, OperatorApp keeps working by progressively falling back to the data it already understands:

1. **Persisted or URL context is applied first.** During construction the app parses `window.location.search` and any embedded context to fill `pageContext`, then writes the values into `state.activeEventId`, `state.activeScheduleId`, and related fields (`applyContextToState`).【F:scripts/operator/app.js†L410-L476】
2. **UI context is refreshed from available inputs.** Each time questions or metadata reload, `updateScheduleContext` looks at the current page context and, if it cannot find an event/schedule pair, consults the most recent display `assignment` pulled from `render/events/<event>/sessions` (or legacy `render/session` as fallback) before updating the banner labels and cached selection keys.【F:scripts/operator/questions.js†L293-L414】
3. **Preflight cache probes precede realtime lookups.** The embed checks `sessionStorage`/`localStorage` for `sos:authPreflightContext`; if present and fresh, it reapplies the cached credential and skips redundant admin provisioning. Without a usable preflight the loader validates membership through the users sheet, calls `ensureAdmin`, and then hydrates itself by reading the RTDB `questions`, `questionStatus/<event>`, and intake metadata branches before wiring realtime subscriptions.【F:scripts/shared/auth-preflight.js†L200-L289】【F:scripts/operator/app.js†L2382-L2448】
4. **Active channel resolution uses multiple fallbacks.** `getActiveChannel` first checks the explicit `state.active*` fields, then tries the derived `state.currentSchedule` or URL-provided schedule key so that a legacy selection remains available even without new session data.【F:scripts/operator/app.js†L483-L503】
5. **Legacy render endpoints remain reachable (deprecated).** The Firebase helpers translate a missing event ID into the historic `render/state` and `render/state/nowShowing` locations, ensuring the app continues reading and writing the global channel when no scoped channel is identified. **Note: These legacy paths are deprecated and will be removed in a future refactoring.**【F:scripts/operator/firebase.js†L48-L70】【F:scripts/shared/channel-paths.js†L1-L112】
6. **Subscriptions adjust automatically.** `refreshChannelSubscriptions` reuses the channel selected in the previous steps, so the operator preview and display controls continue to work against whichever path the fallbacks resolved, including the legacy node.【F:scripts/operator/app.js†L680-L706】

**Important**: The legacy fallback paths (`render/state`, `render/session`, `questionStatus/<uid>`) are maintained for backward compatibility but are scheduled for removal. New code should always use the event-scoped paths (`render/events/<event>/<schedule>/...`, `render/events/<event>/sessions/<uid>`, `questionStatus/<event>/<uid>`).

## 3. Usage guidance for follow-up work

- Treat the tables above as the authoritative list of shared keys when adding new fields—extend them instead of inventing additional ad hoc storage.
- **Use event-scoped paths**: Always use the new event-scoped paths (`render/events/<event>/...`, `questionStatus/<event>/...`) instead of legacy global paths. The legacy paths are deprecated and will be removed.
- **Event ID is required**: The new paths require an `eventId` parameter. Ensure that event context is available before accessing these paths.
- Preserve the fallback flow outlined in §2 so that deployments without the completion flag continue to function; any new signals should layer on top of these helpers instead of bypassing them.
- When introducing a completion flag, consider storing it alongside the existing `render/events/<event>/sessions/<uid>` payload so the same subscription delivers both assignment and completion state, and gate new behavior on the flag while leaving the fallback untouched.

## 4. Migration status and deprecation notice

### Deprecated paths (scheduled for removal)

The following paths are maintained for backward compatibility but will be removed in a future refactoring:

- `render/state` → Use `render/events/<event>/<schedule>/state` instead
- `render/state/nowShowing` → Use `render/events/<event>/<schedule>/nowShowing` instead
- `render/state/sideTelops` → Use `render/events/<event>/<schedule>/sideTelops` instead
- `render/session` → Use `render/events/<event>/sessions/<uid>` instead
- `screens/sessions/<uid>` → Use `render/events/<event>/sessions/<uid>` instead
- `questionStatus/<uid>` → Use `questionStatus/<event>/<uid>` instead

See `docs/firebase-rules-refactoring-impact.md` for detailed migration guidance.
